# P25ML Claude Code 行動規則

## 基本原則

このプロジェクトは巨大シェアード・ワールド小説「PROJECT: 2025 MULTI-LAYER（P25ML）」の
共通世界観データベースと複数シリーズを管理するシステムである。

## 執筆の最優先原則

**1話1話が、それ自体として面白いこと。**

- プロットを進めることには価値がない。読者がその1話を読んで面白いと感じ、続きを読みたいと思うことが最重要
- あらすじの箇条書きを散文化したような文章は禁止。小説を書くこと
- 各話は単独で読んでも読後感が良いこと。場面の余韻、感情の残響、「次が気になる」引き——これらが毎話に存在すること
- 目標文字数は **3,000文字前後**（2,500〜3,500文字の範囲）。短すぎればスカスカになり、長すぎれば間延びする。この範囲で密度を最大化すること
- プロットの都合でシーンを急がない。必要なら1話かけて1つの会話だけを丁寧に書いてよい

## 検索ファースト原則

新規作成や本文生成の前に、必ず以下の順序で検索すること。

1. 対象シリーズの `series.yaml`
2. 対象シリーズの `docs/`
3. 対象シリーズの `plot/`
4. グローバル `db/`
5. `P25ML_MASTER.md`
6. `P25ML_TIMELINE.md`

**推測で既存設定を捏造してはならない。**

## 近似概念チェック

新しい人物・組織・地名・事件・アイテムを作る前に、
**既存に近い概念がないか必ず検索**すること。
類似要素がある場合は、重複作成せず候補提示を優先する。

## クロスオーバー制限

クロスオーバーは自動で実施してはならない。
必要性がある場合でも、**提案のみ**とし、ユーザー確認があるまで確定させてはならない。

## 世界法則の勝手な増設禁止

新しい上位系統、国家級組織、世界規模ルール、宇宙論の更新は独断で行ってはならない。
必要な場合は提案として記述する。

## 実在要素の扱い

- 2025年以降の実在企業、実在サービス、実在個人、実在事件に強く寄せた表現は避けること
- 社会の空気感は現実寄りでよいが、固有要素は架空化すること
- 国名、都道府県名、大都市名、一般的な社会状況、暦は現実準拠でよい
- 企業名、SNS名、サービス名、学校名、病院名、商業施設名、メディア名、行政細部、事件の当事者名は原則すべて架空化すること

## エピソード冒頭のナンバリング規則

各エピソードの本文冒頭（YAMLフロントマターの直後）に、話数とタイトルを記載すること。
形式はシリーズ（著者ペルソナ）ごとに異なる。以下の形式を厳守する。

| シリーズ | 形式 | 例 |
|---------|------|-----|
| SR01 | `第X話「タイトル」` | 第1話「広告つきの朝」 |
| SR02 | `漢数字　タイトル` | 一　封筒 |
| SR03 | `第X話　タイトル`（漢数字） | 第一話　まだ終わってない |
| SR04 | `第X話　タイトル`（算用数字） | 第1話　開業初日の猫 |
| SR05 | `XX　タイトル`（ゼロ埋め数字のみ） | 01　耳鳴りの理由 |

- ナンバリング行の前後に空行を入れること
- タイトルはYAMLフロントマターの `title` フィールドと一致させること

## 執筆時の設定フィードバック原則

執筆中に新たな設定が必要になった場合（地名、技術、組織構造、ルール、アイテム等）、
**設定を自由に追加・拡充してよい**。ただし以下を守ること。

1. 追加した設定は該当する設定ファイル（`docs/local_settings/`、`docs/local_locations/` 等）に必ず反映する
2. 既存設定の具体化・詳細化も同様に設定ファイルへフィードバックする
3. グローバル設定（`db/`）への影響がある場合はユーザー確認を取る
4. 設定ファイルへの反映内容をユーザーに報告する

**本文だけに設定が埋もれた状態を避ける。設定は常にファイルとして管理可能な状態に保つこと。**

## 品質レビュー原則

すべてのエピソードは執筆直後に品質レビューを実施すること。
詳細は `.claudecode/quality_review.md` に定義されている。

- **執筆前**: 必ず `writing_notes.md` と前話の `reviews/` を読む
- **執筆後**: 必ず8項目の品質レビューを実施し、`reviews/` に記録する
- **C判定（不合格）**: 再執筆を行う。問題点を特定してから書き直す
- **知見の蓄積**: レビューで得た教訓を `writing_notes.md` に追記する

**レビューを省略してはならない。品質管理は執筆と同等に重要である。**

## 不明時の原則

不明点がある場合、勝手に断定して埋めず、以下のいずれかに留めること。

- **保留**: 決定を先送りする
- **仮説**: 暫定的な案として記録する
- **提案**: ユーザーに判断を求める

## 正史階層

| 領域 | 正史レベル |
|------|-----------|
| `docs/` | 準正史 |
| `plot/` | 計画 |
| `episodes/` の `canonized` or `published` | **正史** |
| 明示的に `canon: true` の設定 | 正史（本文未登場でも可） |

## 改稿ルール

- 執筆済み本文の重大修正は**禁止**
- 軽微修正は許可されるが、差分要約と記録を必須とする
- retcon（後付け設定変更）は可。ただし、`change_log.md` への明示記録が必須
- 過去正史と矛盾が生じる変更は、勝手に適用せず、**提案止まり**にする

## 削除の原則

削除は原則禁止。廃止・無効化・deprecated化で対応すること。

## 禁止事項一覧

1. 既存正史の無断改変
2. 実在企業や実在サービスへの過度な依存
3. 現実の事件を強く連想させる危険な直写
4. 検索せずに設定を捏造すること
5. クロスオーバーの自動実装
6. 世界法則の独断増設
7. 重大修正を黙って適用すること
8. 削除による整合性解決
